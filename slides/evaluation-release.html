Evaluation And Release · 25-35 minutes

Prove The Model Helps

The final paper should separate useful behavior from attractive demos.

Evaluation

Report the thing users will actually care about.

  • Accuracy, F1, exact match, MRR, pass@k, schema validity, or task completion.
  • Parse failure rate for generated outputs.
  • Latency, memory, and local runtime for the released artifact.

Error Analysis

A strong paper names the failures.

Data

Ambiguity, imbalance, leakage, annotation noise.

Model

Hallucination, format errors, language gaps.

Runtime

Too slow, too large, unstable under quantization.

Release Package

Submit artifacts, not just claims.

  • Model or adapter checkpoint.
  • Model card and dataset card.
  • Training and evaluation config.
  • Local run command.
  • Known limitations.

Ethics

Local does not automatically mean safe.

  • Document data rights and privacy risk.
  • Evaluate harms in the project language, not only English.
  • Do not overclaim general capability from a narrow test.

Final Check

Could another student run it, inspect it, and understand its limits?
